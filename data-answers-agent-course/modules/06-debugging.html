When Things Break

A smoke detector — sometimes it beeps for toast, sometimes for fire. Learn which response means "ask again" vs "stop and route to a human."

Not every beep is a fire

The agent returns one of four statuses in the ApiResponse envelope. Three are intentional design choices — only error means something unexpected broke. Knowing the difference saves you from "fixing" correct behaviour.

Orchestrator

Match + allow + query → answer with provenance

Three intentional exits — all logged via close_record(). A fourth error path exists for infrastructure failures.

🔔
The smoke-detector metaphor

Clarification is toast — annoying but harmless; the system is working. Decline is a real alarm — the question is outside the grounded set or access is denied. Error is a faulty detector — step cap exceeded, token budget blown, or an internal failure. Check the audit log first, always.

ok

Grounding matched one metric → policy allowed → warehouse ran → answer formatted with confidence (e.g. 0.85).

needs_clarification

Grounding found multiple plausible metrics. The agent asks a specific question — it does not guess which metric you meant.

declined

Out of registry, injection flagged, policy deny (empty regions), or classifier marks out_of_scope. Routed to human — no improvised SQL.

error

Infrastructure failure: step cap, token budget, read-only violation, identity misconfiguration. Audit record includes the error detail.

Four scenarios from the UI

The Ask Console ships example scenarios that exercise each path. Use them to verify the agent behaves correctly — not to "jailbreak" it into answering everything.

Clarification

Both total_revenue and net_revenue match — grounding returns ambiguous. UI shows amber "Needs clarification" badge.

Out of scope

Question: "How many unicorns do we have?"

No metric in registry matches → out_of_set → declined with "Out of scope, routed to human."

Policy deny

Question: "What was total revenue last month?"  |  Regions: (empty)

Grounding succeeds but authorize() denies — no_allowed_regions. No warehouse query runs despite a valid metric.

Happy path

Question: "What was total revenue last month?"  |  Regions: US, EU

Single metric match → policy allow → parameterized template executes → answer with source total_revenue and confidence 0.85.

Debugging with AI — what do you tell it?

When pairing with an AI coding assistant on this codebase, anchor it to the five non-negotiables. Wrong instructions produce confidently wrong architecture.

1. The agent keeps guessing SQL for out-of-scope questions. What do you tell the AI assistant?

2. User u2 with empty regions gets denied on a valid revenue question. First debugging step?

3. "What was revenue last month?" returns needs_clarification. Is this a bug?

4. A declined request returns no answer text. The user says "it's broken." What do you check first?

🛡️
The five non-negotiables — your debugging compass

1. Grounding — vetted metrics only; clarify or decline, never guess.
2. Governance — access control in the data layer, never in prompts.
3. Identity — principal flows end-to-end; agent acts as the user.
4. Read-only — mutation structurally impossible; no DML/DDL anywhere.
5. Auditable — every request reconstructable: who, what, when, allow/deny, bytes, latency.
