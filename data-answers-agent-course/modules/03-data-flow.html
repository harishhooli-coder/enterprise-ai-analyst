The relay race

When a business user asks a question, the API hands off a single baton: request_id. Every actor in the pipeline touches that same ID — logging it in the audit trail, checking guardrails, resolving grounding, and closing the loop. No runner improvises; each does one job and passes the baton forward.

🏃
Runner 1 — Audit opens the race

open_record(request_id, …) stamps who asked and the raw question before anything else runs.

🏃
Runners 2–7 — Guard, classify, ground, authorize, query, format

Each step calls step(name), incrementing a counter guarded by the step cap.

🏁
Runner 8 — Audit closes the race

close_record(request_id, status, …) seals latency, outcome, and any decline reason — success or refusal.

Where the race starts in code

The orchestrator opens the audit record before guardrails run — so even a declined request leaves a reconstructable trail.

orchestrator.py try: # 1. audit-open open_record( request_id, request.user_principal.model_dump(), request.question, ) step("audit_open")
In plain English
Line 51 — Enter a try block so any failure still closes the audit record.
Lines 53–57 — Log request_id, the user principal, and the exact question. This is the baton handoff to Audit.
Line 58 — Count step 1 toward the step cap. Next up: scan_input checks the question for injection or secrets.

The nine-step loop

Every happy-path request walks these steps in order. Any step can short-circuit to declined, needs_clarification, or error — but the baton always returns to Audit for close_record.

1
audit_open

Open the audit record with principal and question.

2
scan_input

Guardrails scan for injection attempts and leaked secrets.

3
classify

Router classifies intent — in scope, out of scope, or suspicious.

4
grounding_resolve

Match the question to a verified metric template from the registry.

5
policy_authorize

Policy gate allows or denies access based on the user principal — not the prompt.

6
identity_mint

Mint an execution context so the warehouse runs as the calling user.

7
warehouse_query

Execute the parameterized template with maximum_bytes_billed enforced.

8
reason_format

Frontier model formats row values into business language — never SQL.

9
redact_output → audit_close

Strip sensitive leaks, then seal the audit record with final status and latency.

⏱
Bounded loops — by design

Two hard ceilings protect cost and runaway agents: the step cap (each step() call increments a counter; exceeding it raises StepCapExceededError) and the token budget. An unbounded agent loop is an unbounded bill — the skeleton forbids both.
