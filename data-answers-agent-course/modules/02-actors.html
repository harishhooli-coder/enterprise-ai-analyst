Meet the Cast

Like a film crew — every role has one job, and nobody else does it for them.

One question, five specialists

When you clicked Ask in Module 1, your question did not magically become a number. It passed through a small cast of components — each with a single responsibility.

Think of a movie set: the director does not hold the camera, and the camera operator does not write the script. Same idea here. If you know who does what, reading the repo stops feeling like alphabet soup.

🎬
Separation of concerns

The UI never queries BigQuery. The warehouse never picks which metric to use. The orchestrator never renders HTML. Each piece stays in its lane so the system stays safe and auditable.

The five roles on set

Hover or tap each card. These names show up in file paths, logs, and error messages — worth memorizing like character names in a show.

🖥

AskConsole

The Test Console UI in data-answers-ui. Collects your question and user info, shows the answer card, and never touches the database.

⚡

main.py /ask

The FastAPI front door. Receives POST /ask, stamps a tracking ID, delegates to the orchestrator, and records metrics.

🎯

Orchestrator

The director in app/loop/orchestrator.py. Decides: answer, clarify, or decline. Calls grounding and warehouse tools through the policy gate.

📚

GroundingService

The librarian. Matches your words to a vetted metric or verified query in registry.yaml — never improvises SQL from scratch.

🏭

Warehouse / BigQuery

The read-only data runner in tools/warehouse.py. Executes parameterized templates against BigQuery with a byte cap — no writes, ever.

Question flows UI → API → Orchestrator → Grounding → Warehouse → back up the chain.

What main.py does at the front door

When the chat reaches main.py /ask, this is the code that runs. Notice how little it does itself — mostly handoff and bookkeeping.

main.py 
@app.post("/ask", response_model=ApiResponse[AnswerPayload])
async def ask(body: AskRequest) -> ApiResponse[AnswerPayload]:
    request_id = str(uuid.uuid4())
    result = orchestrator.handle(body, request_id)

    records = get_audit_sink().records_for(request_id)
    record = records[0] if records else {}
    metrics_collector.record_request(
        status=result.status,
        latency_ms=record.get("latency_ms"),
        bytes_scanned=record.get("bytes_scanned"),
        decline_reason=result.decline_reason,
    )
    return result
Plain English
Lines 43–44 — Register POST /ask as the question endpoint. FastAPI validates the incoming JSON into an AskRequest object before your function body runs — that validation layer is part of the framework’s middleware stack.
Line 45 — Mint a fresh UUID for this request. Every log line for this question shares that ID.
Line 46 — Pass the question to the orchestrator.handle — the API route itself does not pick metrics or run SQL.
Lines 48–55 — Pull audit details (latency, bytes scanned) and feed them into metrics_collector.record_request so operators can track cost and health.
Line 56 — Return the orchestrator result wrapped in the standard ApiResponse envelope back to AskConsole.
