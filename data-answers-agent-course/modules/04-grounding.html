Grounding & Warehouse

Think of a professional kitchen: the chef never invents a dish from scratch. They pick an approved recipe card, gather the listed ingredients, and follow the steps exactly.

Why grounding matters

The Data-Answers Agent serves non-technical business users who cannot verify SQL. The primary path is never free-form text-to-SQL. Every answer must trace to a defined metric or verified query in registry.yaml.

⚠
Confidently wrong is worse than no answer

If a question does not map to the grounded set, the agent returns declined and routes to a human. It never improvises SQL to "be helpful." A polished wrong number destroys more trust than an honest "I cannot answer that yet."

📋

registry.yaml

The recipe box — metric definitions, aliases, and parameterized query templates loaded at startup.

🔍

grounding_service.py

The sous chef — matches the user question to a recipe card, extracts parameters like month, and returns ambiguous or out-of-set when no safe match exists.

🏭

warehouse.py

The kitchen line — executes only approved templates against BigQuery, enforces read-only, injects the row filter, and bills a bounded scan.

Together these three modules implement the first non-negotiable: GROUNDING. The semantic layer substrate may grow beyond flat YAML later — but the contract stays the same: vetted templates only.

Recipe card: total_revenue

Each metric in the registry carries a template with placeholders. Notice @month is a bound parameter, while {region_filter} is resolved server-side from the user principal — never from the question.

registry.yaml · lines 15–19 template: | SELECT SUM(amount) AS total_revenue FROM `{project}.{dataset}.sales` WHERE month = @month AND {region_filter}
In plain English
Line 15 — Start a multi-line SQL template block for the total_revenue metric.
Line 16 — Sum the amount column; alias the result as total_revenue.
Line 17 — Read from the sales table (project and dataset come from config, not user text).
Line 18 — Filter to one calendar month via the @month parameter extracted from the question.
Line 19 — AND the row filter injected by warehouse.py from allowed_regions — governance lives in the data layer.

Kitchen safety checks

Before any template hits BigQuery, the warehouse tool verifies it is read-only and builds the region predicate from the execution context — structural enforcement, not prompt instructions.

warehouse.py · lines 27–38 def assert_readonly(sql: str) -> None: """Reject templates containing DML/DDL verbs.""" if WRITE_VERBS.search(sql): raise ReadOnlyViolationError("Template contains write verb") def region_filter_clause(regions: list[str]) -> str: """Build a server-side SQL fragment stub for row-level region filtering.""" if not regions: return "1=0" quoted = ", ".join(f"'{region.replace(chr(39), '')}'" for region in regions) return f"region IN ({quoted})"
In plain English
assert_readonly — Scan the template for INSERT, UPDATE, DELETE, DDL verbs. Mutation is structurally impossible.
Empty regions — Return 1=0 so the query returns no rows (safe deny), linking to policy no_allowed_regions upstream.
region_filter_clause — Build region IN ('EMEA', 'APAC') from the principal allowed list. The user question never supplies this fragment.
