01

Ask a Question

Like asking a reference librarian for a pre-approved fact sheet — not for a guess.

A librarian, not an improviser

Imagine you walk up to a reference desk and ask: What was total revenue last month? You do not want the librarian to invent numbers from memory. You want them to pull a vetted fact sheet from a trusted shelf.

That is what the Data-Answers Agent does for business users. You type a plain-English question in plain language. The system looks up a pre-approved answer path — never free-form SQL improvisation — and returns a number you can trust.

Two pieces work together: data-answers-ui (the Test Console you click in) and data-answers-agent (the Python backend that does the heavy lifting).

What main.py does at the front door

When the chat reaches main.py /ask, this is the code that runs. Notice how little it does itself — mostly handoff and bookkeeping.

main.py 
@app.post("/ask", response_model=ApiResponse[AnswerPayload])
async def ask(body: AskRequest) -> ApiResponse[AnswerPayload]:
    request_id = str(uuid.uuid4())
    result = orchestrator.handle(body, request_id)

    records = get_audit_sink().records_for(request_id)
    record = records[0] if records else {}
    metrics_collector.record_request(
        status=result.status,
        latency_ms=record.get("latency_ms"),
        bytes_scanned=record.get("bytes_scanned"),
        decline_reason=result.decline_reason,
    )
    return result
Plain English
Lines 43–44 — Register POST /ask as the question endpoint. FastAPI validates the incoming JSON into an AskRequest object before your function body runs — that validation layer is part of the framework’s middleware stack.
Line 45 — Mint a fresh UUID for this request. Every log line for this question shares that ID.
Line 46 — Pass the question to the orchestrator.handle — the API route itself does not pick metrics or run SQL.
Lines 48–55 — Pull audit details (latency, bytes scanned) and feed them into metrics_collector.record_request so operators can track cost and health.
Line 56 — Return the orchestrator result wrapped in the standard ApiResponse envelope back to AskConsole.

Follow the Data

Trace one question through orchestrator.handle — a relay race where the request_id baton passes from runner to runner until the answer crosses the finish line.

The relay race

When a business user asks a question, the API hands off a single baton: request_id. Every actor in the pipeline touches that same ID — logging it in the audit trail, checking guardrails, resolving grounding, and closing the loop. No runner improvises; each does one job and passes the baton forward.

🏃
Runner 1 — Audit opens the race

open_record(request_id, …) stamps who asked and the raw question before anything else runs.

🏃
Runners 2–7 — Guard, classify, ground, authorize, query, format

Each step calls step(name), incrementing a counter guarded by the step cap.

🏁
Runner 8 — Audit closes the race

close_record(request_id, status, …) seals latency, outcome, and any decline reason — success or refusal.

The nine-step loop

Every happy-path request walks these steps in order. Any step can short-circuit to declined, needs_clarification, or error — but the baton always returns to Audit for close_record.

⏱
Bounded loops — by design

Two hard ceilings protect cost and runaway agents: the step cap (each step() call increments a counter; exceeding it raises StepCapExceededError) and the token budget. An unbounded agent loop is an unbounded bill — the skeleton forbids both.

04

Grounding & Warehouse

Think of a professional kitchen: the chef never invents a dish from scratch. They pick an approved recipe card, gather the listed ingredients, and follow the steps exactly.

Why grounding matters

The Data-Answers Agent serves non-technical business users who cannot verify SQL. The primary path is never free-form text-to-SQL. Every answer must trace to a defined metric or verified query in registry.yaml.

⚠
Confidently wrong is worse than no answer

If a question does not map to the grounded set, the agent returns declined and routes to a human. It never improvises SQL to "be helpful." A polished wrong number destroys more trust than an honest "I cannot answer that yet."

📋

registry.yaml

The recipe box — metric definitions, aliases, and parameterized query templates loaded at startup.

🔍

grounding_service.py

The sous chef — matches the user question to a recipe card, extracts parameters like month, and returns ambiguous or out-of-set when no safe match exists.

🏭

warehouse.py

The kitchen line — executes only approved templates against BigQuery, enforces read-only, injects the row filter, and bills a bounded scan.

Together these three modules implement the first non-negotiable: GROUNDING. The semantic layer substrate may grow beyond flat YAML later — but the contract stays the same: vetted templates only.

Recipe card: total_revenue

Each metric in the registry carries a template with placeholders. Notice @month is a bound parameter, while {region_filter} is resolved server-side from the user principal — never from the question.

registry.yaml · lines 15–19 template: | SELECT SUM(amount) AS total_revenue FROM `{project}.{dataset}.sales` WHERE month = @month AND {region_filter}
In plain English
Line 15 — Start a multi-line SQL template block for the total_revenue metric.
Line 16 — Sum the amount column; alias the result as total_revenue.
Line 17 — Read from the sales table (project and dataset come from config, not user text).
Line 18 — Filter to one calendar month via the @month parameter extracted from the question.
Line 19 — AND the row filter injected by warehouse.py from allowed_regions — governance lives in the data layer.

Kitchen safety checks

Before any template hits BigQuery, the warehouse tool verifies it is read-only and builds the region predicate from the execution context — structural enforcement, not prompt instructions.

warehouse.py · lines 27–38 def assert_readonly(sql: str) -> None: """Reject templates containing DML/DDL verbs.""" if WRITE_VERBS.search(sql): raise ReadOnlyViolationError("Template contains write verb") def region_filter_clause(regions: list[str]) -> str: """Build a server-side SQL fragment stub for row-level region filtering.""" if not regions: return "1=0" quoted = ", ".join(f"'{region.replace(chr(39), '')}'" for region in regions) return f"region IN ({quoted})"
In plain English
assert_readonly — Scan the template for INSERT, UPDATE, DELETE, DDL verbs. Mutation is structurally impossible.
Empty regions — Return 1=0 so the query returns no rows (safe deny), linking to policy no_allowed_regions upstream.
region_filter_clause — Build region IN ('EMEA', 'APAC') from the principal allowed list. The user question never supplies this fragment.

Checkpoint 1 — Policy Gate

Every tool call passes through PEP → PDP authorization in gate.py. The orchestrator calls authorize(principal, metric_id) after grounding resolves — if the principal has no allowed regions, the request is denied before BigQuery ever runs.

gate.py 
def authorize(
    principal: UserPrincipal,
    resource: str,
    metadata: Optional[dict[str, Any]] = None,
    *,
    audit_callback: Optional[AuditCallback] = None,
) -> AllowDeny:
    """Allow if principal has at least one allowed region; otherwise deny."""
    meta = metadata or {}

    if not principal.allowed_regions:
        decision = AllowDeny(allowed=False, reason="no_allowed_regions")
    else:
        decision = AllowDeny(allowed=True, reason="skeleton_allow")

    if audit_callback is not None:
        audit_callback("policy_decision", principal, resource, meta, decision)

    return decision
In plain English
Line 21: Check whether the principal has any allowed regions.
Lines 24–25: Empty regions → deny immediately with reason no_allowed_regions.
Lines 26–27: Skeleton allows if regions exist (real PDP replaces this).
Lines 29–30: Every decision is logged to audit before returning.
Line 32: Return AllowDeny — orchestrator stops if not allowed.
🚫
Deny path

Principal with empty allowed_regions → declined_response("Access denied: no_allowed_regions"). No warehouse query executes.

✓
Allow path

Principal passes → identity broker mints execution context → parameterized template runs with row filter derived from principal.

Checkpoint 2 — Guardrails

Guardrails run on input before the agent loop and on output before the answer reaches the user. They catch injection attempts and strip leaked schema from responses.

guardrails.py 
def scan_input(question: str) -> ScanResult:
    """Flag injection markers or embedded secrets in the user question."""
    for pattern in INJECTION_PATTERNS:
        if pattern.search(question):
            return ScanResult(flagged=True, reason="injection_pattern_detected")

    for pattern in SECRET_PATTERNS:
        if pattern.search(question):
            return ScanResult(flagged=True, reason="secret_pattern_detected")

    return ScanResult(flagged=False, reason=None)
In plain English
Lines 32–34: Scan for injection patterns — "ignore previous instructions", ```sql, ; DROP TABLE, etc.
Lines 36–38: Scan for embedded secrets — AWS keys, private keys.
Line 40: Clean input → proceed to classification and grounding.
📥

scan_input (lines 30–40)

Runs first in orchestrator. Flagged input → immediate decline, no grounding, no SQL. Reason logged to audit.

📤

redact_output (lines 43–50)

After the frontier model formats the answer, regex patterns replace leaked SQL keywords, backtick table refs, and schema paths with [redacted].

🔒

PII & secrets

Skeleton uses regex stubs. Hardening wires Presidio and expanded rule catalogs — but the boundary is already wired.

⚖️
Governance lives in the data layer — not in prompts

Never write a prompt that says "only show data the user is allowed to see." That is not security — it is hope. Access control is structural: RLS in BigQuery, policy gate before tool calls, row filters derived from the principal. Prompts format answers; they do not enforce permissions.

When Things Break

A smoke detector — sometimes it beeps for toast, sometimes for fire. Learn which response means "ask again" vs "stop and route to a human."

Not every beep is a fire

The agent returns one of four statuses in the ApiResponse envelope. Three are intentional design choices — only error means something unexpected broke. Knowing the difference saves you from "fixing" correct behaviour.

Orchestrator
handle(request)
✓

ok
Match + allow + query → answer with provenance

?

needs_clarification
Ambiguous metric — ask, don't guess

✕

declined
Out of set, policy deny, or flagged input

Three intentional exits — all logged via close_record(). A fourth error path exists for infrastructure failures.

🔔
The smoke-detector metaphor

Clarification is toast — annoying but harmless; the system is working. Decline is a real alarm — the question is outside the grounded set or access is denied. Error is a faulty detector — step cap exceeded, token budget blown, or an internal failure. Check the audit log first, always.

ok

Grounding matched one metric → policy allowed → warehouse ran → answer formatted with confidence (e.g. 0.85).

needs_clarification

Grounding found multiple plausible metrics. The agent asks a specific question — it does not guess which metric you meant.

declined

Out of registry, injection flagged, policy deny (empty regions), or classifier marks out_of_scope. Routed to human — no improvised SQL.

error

Infrastructure failure: step cap, token budget, read-only violation, identity misconfiguration. Audit record includes the error detail.

🛡️
The five non-negotiables — your debugging compass

1. Grounding — vetted metrics only; clarify or decline, never guess.
2. Governance — access control in the data layer, never in prompts.
3. Identity — principal flows end-to-end; agent acts as the user.
4. Read-only — mutation structurally impossible; no DML/DDL anywhere.
5. Auditable — every request reconstructable: who, what, when, allow/deny, bytes, latency.
